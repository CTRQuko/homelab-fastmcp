# runtime-issues.md — Registro de errores de ejecución

Errores reales ocurridos durante sesiones de operación con cualquier
LLM/CLI cliente de mimir-mcp (Claude Code, OpenCode, etc.). No teoría —
solo incidentes concretos con causa raíz y fix.

## Cómo se alimenta este archivo

Dos fuentes complementarias:

### 1. Manual (operador escribe directamente)

Cuando ves un fallo y quieres anotar contexto narrativo (qué hacías,
qué esperabas, qué pasó). Convención de entrada abajo.

### 2. Auto-generado desde audit.log

Cualquier error de tool call de mimir queda registrado por
`core/audit.py` en `config/audit.log` (JSONL) con `error_message`,
`client`, `args_sanitized`. El script
`scripts/audit_to_runtime_issues.py` lee ese audit log y genera
*skeleton entries* aquí (con `<pendiente>` en causa/fix/prevención
para que tú las completes).

**Auto-generado en sesiones Claude Code** (vía hook `Stop` en
`~/.claude/settings.json`):

```json
"hooks": {
  "Stop": [
    {
      "hooks": [
        {
          "type": "command",
          "command": "python C:/homelab/mcp-servers/mimir-mcp/scripts/audit_to_runtime_issues.py --since-session-start --append-to C:/homelab/mcp-servers/mimir-mcp/docs/operator-notes/runtime-issues.md --session-tag claude-${CLAUDE_SESSION_ID:-stop}"
        }
      ]
    }
  ]
}
```

Al cerrar la sesión Claude Code, el hook ejecuta el script y anexa
errores nuevos como entries auto.

**OpenCode (sin hooks nativos) — manual**:

```bash
# Al terminar sesión OpenCode:
python C:/homelab/mcp-servers/mimir-mcp/scripts/audit_to_runtime_issues.py \
       --since "2 hours ago" \
       --session-tag "opencode-$(date +%Y%m%d-%H%M)"
```

Cron job periódico para OpenCode → diferido a v0.6.0 hasta validar
flujo Claude Code primero.

**Comandos útiles del script**:
- `--dry-run`: imprime el bloque sin tocar el `.md`
- `--since "30 minutes ago"`, `"2 hours ago"`, `"1 day ago"`, ISO datetime, unix ts
- `--audit-log <path>`: override (default: `MIMIR_AUDIT_LOG` env o `config/audit.log`)
- `--session-tag <tag>`: distintivo en cada entry (default: `auto-YYYYMMDD-HHMM`)

Backup del `.md` se hace en `runtime-issues.md.bak` antes de cada append.
El script es **append-only** — nunca sobreescribe contenido existente.

## Convención para añadir entradas (manual o completar skeleton)

```
### [YYYY-MM-DD HHMM] Título corto (sesión: nombre-sesión)
- agente: claude-code | opencode | codex | ...
- **Síntoma**: qué falló exactamente (mensaje de error o comportamiento)
- **Causa**: por qué falló
- **Fix aplicado**: qué se hizo para resolverlo
- **Prevención**: cómo evitarlo en el futuro
```

Una entrada por incidente. Si el mismo error ocurre en otra sesión, añadir una línea
`Reincidencia: YYYY-MM-DD` al final de la entrada existente, no duplicar.

Las entries auto-generadas vienen con `<pendiente>` en Causa/Fix/Prevención
y un comentario HTML `<!-- auto-generated by audit-to-runtime-issues.py -->`
delimitando el bloque. Cuando tengas tiempo, completa los pendientes y
el comentario puede quedarse o quitarse — solo es marca de origen.

### [2026-05-03] regex con `|` interpretado como pipe shell por wrapper (sesión: ajustandose-a-mi-red)

- **Causa**: El wrapper invoca el comando reconstruyendo argumentos. Sin quoting fuerte, los `|` del regex se reinterpretan como pipes de shell durante la expansión del wrapper, ejecutando `listeningMode`, `domain`, `upstreams`, `hosts` como comandos separados.

Asegura ORDEN del servicio en OpenRC, pero `tailscaled` levanta la interfaz `tailscale0` de forma asíncrona después de que el servicio reporte "started".   [date] reiniciando pihole para rebind dnsmasq
  ```
  [date] pihole restart completado
  ```

### [2026-05-03] Bridge Docker SNAT impide remote_ip filter en Caddy (sesión: ajustandose-a-mi-red)
- **Síntoma**: Caddyfile con `@tailscale { remote_ip 100.64.0.0/10 }` rechaza con 403 a clientes Tailscale legítimos. Caddy ve la IP del cliente como `172.18.0.1` (gateway del bridge Docker) en lugar de la IP TS real.
- **Causa**: Cuando Caddy corre en una red Docker bridge con `ports: ["100.69.126.35:443:443"]`, el `docker-proxy` o `iptables NAT` (con `userland-proxy=true` por defecto) hace SNAT de la IP fuente del cliente a la IP del bridge. Caddy ve solo la IP del bridge — el filtro `remote_ip` rompe.
- **Fix aplicado**: Cambiar Caddy a `network_mode: host`. Backends (Vaultwarden, Arcane) publican puertos solo en `127.0.0.1:<puerto>` y Caddy los alcanza por loopback. En el Caddyfile añadir `default_bind 100.69.126.35` para que Caddy bind exclusivo a IP Tailscale del host (no a 0.0.0.0).
- **Prevención**: Cualquier proxy/firewall que dependa de `remote_ip` debe correr en `network_mode: host` o usar `userland-proxy: false` en el daemon Docker. Si se necesita `remote_ip` con bridge, configurar `trusted_proxies` y exigir cabecera X-Forwarded-For del proxy upstream (no aplica si Caddy es el frontal).

### [2026-05-03] sudo -n docker rechaza pero usuario en grupo docker (sesión: ajustandose-a-mi-red)
- **Síntoma**: `ssh hetzner 'sudo -n docker exec ...'` falla con `sudo: a password is required`, aunque el comando es estándar de operación.
- **Causa**: `sudo -n docker exec` no está en NOPASSWD de claude (solo `docker ps/images/inspect/logs/stats/start/stop/restart` están). Pero `claude` está en el grupo `docker`, lo que permite ejecutar `docker` directamente sin sudo.

### [2026-05-03] sudoers regeneration self-lockout cuando script viejo elimina su propia regla (sesión: ajustandose-a-mi-red)
- **Síntoma**: Tras `sudo bash /opt/claude-scripts/sudoers-claude.sh` ejecutado con la versión vieja del script en pve y pve2, los siguientes intentos del mismo comando fallan con `sudo: a password is required`. La regla `NOPASSWD: /bin/bash /opt/claude-scripts/sudoers-claude.sh *` desaparece del sudoers regenerado.
- **Causa**: La versión vieja del script (`75dccfbc...`) NO incluía la sección `## === SCRIPTS DESPLIEGUE` (líneas 297-303 en versión nueva `9c5e9e3a...`) que genera las reglas NOPASSWD para los propios scripts del repo. Al regenerar `/etc/sudoers.d/claude` con script viejo, esas reglas desaparecen del sudoers — auto-lockout. El claude-user pierde la capacidad de re-ejecutar el script (incluso si después se sube la versión nueva al sistema). El archivo en `/opt/claude-scripts/sudoers-claude.sh` puede ser propiedad de `claude:claude` y por tanto escribible directamente con `scp` (sin sudo) — esto es lo que confunde, porque el archivo SÍ se actualiza pero no se puede ejecutar.
- **Fix aplicado**: Operador ejecutó `bash /opt/claude-scripts/sudoers-claude.sh apply` desde Shell del Web UI Proxmox del nodo afectado (root directo, sin sudo). Tras esa ejecución, el sudoers nuevo incluye la regla `bash sudoers-claude.sh *` y a partir de ahí Claude puede regenerar autónomamente.
- **Prevención**:
  1. NUNCA regenerar sudoers con un script más antiguo que el que está actualmente en producción. Verificar `md5sum /opt/claude-scripts/sudoers-claude.sh` vs el del repo ANTES de ejecutar.
  2. Antes de hacer `scp` del nuevo script al nodo, verificar que la versión nueva incluye la sección `## === SCRIPTS DESPLIEGUE` y la línea `for script in sudoers-claude.sh deploy-wrapper.sh api-permissions.sh`. Sin eso, el regenerar elimina las reglas auto-bootstrap.
  3. Si ocurre lock-out: bootstrap manual una vez vía Web UI Proxmox (Shell del nodo es root directo, no usa sudo), después auto.
  4. El truco de sintaxis: la regla NOPASSWD `bash X *` requiere matchear con AL MENOS un argumento. Invocar como `bash X apply` (o cualquier string) — `bash X` sin argumentos NO matchea y pide password. Esto es ortogonal al lock-out pero contribuyó a la confusión inicial.

### [2026-05-03] MCP server transient disconnect / schema mismatch (sesión: ajustandose-a-mi-red)
- **Síntoma**: Errores tipo `MCP error -32602: Input validation error` o desconexión silenciosa de un tool MCP que estaba disponible al inicio de la sesión.
- **Causa**: Dos modos:
  1. **Schema mismatch (-32602)**: tipos JSON enviados no coinciden con el schema declarado por el server (caso `mem_get_observation` con string).
  2. **Transient disconnect**: el server MCP cae o reinicia durante una sesión larga; tools quedan listadas pero invocaciones fallan hasta reconexión.
- **Fix aplicado**: En caso 1, ajustar tipo del argumento. En caso 2, reintentar tras unos segundos; si persiste, reiniciar el server MCP afectado o el cliente Claude.
- **Prevención**: (a) Validar tipos antes de llamar tools sensibles a schema. (b) En sesiones largas, hacer health checks ocasionales del MCP crítico (engram, mimir-mcp). (c) Si un MCP es crítico para el flujo, no asumir que sigue conectado: tener fallback (`mem_search` directo si `mem_get_observation` falla, lectura directa de archivos si el MCP de inventario cae, etc.).

## Patrones recurrentes

| Patrón | Síntoma típico | Raíz común | Mitigación |
|--------|---------------|------------|-----------|
| **Script drift nodo vs repo** | Comportamiento inesperado tras re-deploy | Scripts en `/opt/claude-scripts/` no se auto-actualizan | `diff` nodo vs repo antes de ejecutar; `scp` al cambiar el repo |
| **sudo -S falla en no-NOPASSWD** | `Sorry, try again` con contraseña correcta | Contraseña `claude-sudo.key` no es universal; cada nodo puede tener distinta | Verificar con `sudo -S whoami` antes de asumir que funciona |
| **bash history expansion con `!`** | Patrón `sed`/`grep` destruido silenciosamente | `!` en double quotes activa histexpand | Siempre comillas simples cuando el string contiene `!` |
| **Wrapper conf desactualizado** | Comandos bloqueados que deberían funcionar | `deploy-wrapper.sh` usa el perfil del nodo, no del repo | `scp` perfiles al nodo antes de `deploy-wrapper.sh` |
| **Alias SSH faltante para jandro** | `Permission denied` al usar `-l jandro` | `~/.ssh/config` usa `claude_key` para todos los hosts PVE | Añadir aliases `pve-jandro`, `pve2-jandro`, `pve3-jandro` con `homelab_key` |
| **Nombres de servicio systemd vs OpenRC** | `service not found` en Alpine | Nombres como `pihole-FTL` son systemd; Alpine usa `pihole` | `rc-service -l` primero en nodos Alpine |
| **Parallel batch cancelado por un fallo** | `Cancelled: parallel tool call` | Claude cancela todo el batch si una llamada falla | Separar comandos incertidumbre; añadir `\|\| true` donde proceda |
| **jq no disponible en Git Bash Windows** | `jq: command not found` (exit 127) | Git Bash de Windows no incluye jq por defecto | Usar `python -c "import sys,json; ..."` para parsing JSON desde Windows; jq solo cuando se ejecute en host Linux remoto |
| **Regex con `\|` como pipe shell en wrapper** | `wrapper: line N: <token>: command not found` | Wrapper reconstruye argumentos sin preservar quoting fuerte; `\|` del regex se reinterpreta como pipe | Patrones sin `\|` o greps separados; usar `grep -e p1 -e p2` en su lugar |
| **find / en LXC unprivileged ruidoso** | Decenas de `find: /dev/.lxc/...: Permission denied` | `/dev/.lxc/sys/*` es bind-mount restrictivo en LXC unprivileged | Acotar `find` a `/etc`, `/usr`, etc.; nunca `find /` sin scope |
| **MCP tool schema mismatch** | `MCP error -32602: Expected X, received Y` | Schema JSON del tool exige tipo concreto (e.g. `number`) y la invocación pasa otro (`string`) | Verificar schema antes de invocar; pasar tipos literales correctos (id sin comillas para `number`) |
| **Race condition de servicios al boot OpenRC** | Servicio dependiente arranca antes que el proveedor; bind a interfaz inexistente; daemon en estado inválido sin auto-rebind | Falta `need <servicio>` en `depend()` del init script; OpenRC arranca en orden alfabético/runlevel sin dependencia explícita | Añadir `rc_need="<dep>"` en `/etc/conf.d/<servicio>` o `depend() { need <dep>; }` en `/etc/init.d/<servicio>` |
| **MCP server transient disconnect** | Tool listado pero invocación falla; sesión larga | Server MCP cae/reinicia silenciosamente | Health check ocasional; fallback a alternativas (`mem_search` si `mem_get_observation` falla, lectura directa si MCP de inventario cae) |
| **DNS debug sin verificar resolver real** | Parches a un Pi-hole funcionan pero MagicDNS no resuelve igual | El cliente Tailscale consulta a OTRO resolver (split-DNS distinto) y los parches van al sitio equivocado | Antes de tocar cualquier resolver: `curl /api/v2/tailnet/$t/dns/split-dns` + `dns/nameservers`. Comprobar a quién pregunta el cliente afectado |
| **Post-create LXC sin wrapper / sudoers** | `pct exec <CTID> -- claude-wrapper ...` cuelga, timeouta o devuelve "command not found" | El LXC se creó pero NO se ejecutó el checklist post-create | SIEMPRE tras `pct create`: `deploy-wrapper.sh <CTID> <perfil>` + `sudoers-claude.sh`. Verificar con `pct exec <CTID> -- /usr/local/bin/claude-wrapper echo ok`. Documentado en `~/.claude/rules/homelab-scripts.md` |
| **Cierre de sesión con estado reportado, no verificado** | Próxima sesión hereda diagnóstico desactualizado y rebota sobre él | El cierre describe "lo planeado" o "lo intentado" en vez del estado actual del sistema | Antes de cerrar: `docker ps` en cada sede tocada, `validate.sh` con output guardado, NO confiar en "lo que se intentó" como si fuera "lo que está" |
| **Iterar 5+ veces contra el síntoma sin re-validar la hipótesis base** | Tarea que debería tomar 1h se va a 4h con misma causa raíz repetida | Tras 2-3 intentos fallidos, el asistente sigue pulsando el mismo botón en lugar de parar y verificar el modelo mental | Regla 3-strikes: si tras 3 intentos un fix no avanza, parar y verificar la hipótesis base con consulta primaria (API, ssh directo, no memoria) antes de continuar |
| **Engram backend split — IDs no resolvibles** | `mem_get_observation 9614` → "not found" pese a aparecer en system-reminder | Los IDs `SXXX` y `9XXX` del CMEM banner son de claude-mem (otro backend), no de engram (IDs ~3XX) | Para detalle de obs SXXX/9XXX: `claude-mem:mem-search` o `mcp__plugin_claude-mem_mcp-search__get_observations`. Engram tiene IDs propios (3XX) |
| **Cat de secret en host remoto (path Windows)** | `sudo: a password is required` o `cat: No such file` | El comando `ssh host '...$(cat C:/homelab/.config/secrets/x.key)...'` ejecuta el `cat` en el host REMOTO Linux donde el path Windows no existe | Leer el secret en local primero: `PASS=$(cat /c/homelab/.config/secrets/x.key)` y pasar a ssh por stdin: `ssh host "..." <<< "$PASS"`, NO interpolar la variable en el shell remoto |
